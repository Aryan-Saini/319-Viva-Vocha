Executive Summary

        This document presents our design of Aunt Leah’s content management system designed to support the employees of Aunt Leah to manage their programs and track the progress of their clients through their respective programs. In order to accomplish this within a constrained development window, we have decided on a client server monolithic architecture, sacrificing on future scalability.

Design Decisions for System Design

  1. Monolithic vs Microservices architecture

  1. Decision - Monolithic architecture
  2. Rationale
  1. The maximum assumed scale as per our assumptions is a maximum of 10 concurrent users and hold fewer than 2000 clients, a monolithic architecture would be sufficient to handle this.
  2. A 6-8 week development window prioritises velocity above all. Monolithic architectures reduce system complexity
  3. No inter-service communication reduces overhead
  4. We only need one set of CI/CD pipelines instead of one for each microservice
  5. Shared TypeScript types force frontend and backend to be consistent
  1. Trade offs
  1. Lost
  1. In the interest of speed, we have sacrificed scalability and future migration to a microservice architecture would be harder
  2. The application bundle would be larger, even though it is optimised by code splitting
  1. Gained
  1. A less complex monolithic application saves up dev time (estimated 20+ hours), especially come deployment

  1. Consistency vs Availability

  1. Decision - Strong Consistency over Availability
  2. Rationale
  1. For the audit trail requirement, inconsistent logs (stale data) could lead to the IT Team missing unauthorised use
  2. For editing case notes or client details, we would need immediate change. Otherwise, workers could see stale data during meetings
  3. Inconsistency in session token invalidations could lead to unauthorized access to client data
  4. Since concurrent edits are out of scope and a maximum of 10 concurrent users are assumed, our need for horizontal scaling isn’t pressing and a single region hosted AWS instance (around 99.9% availability) would suffice, allowing us to focus more on consistency and ACID compliance
  1. Tradeoffs
  1. Lost
  1. 99.99% uptime by not choosing a distributed region deployment using CockroachDB
  2. We have to use database write locks, increasing latency for edit operations
  1. Gained
  1. Although, this has increased the complexity of our API layers as each operator is now wrapped in a begin transaction loop, we have saved a lot of time in the DevOps stage by not using distributed deployments across regions
  2. Audit trail compliance
  3. Immediate synchronous updates for client profiles and case notes
  4. Consistency in session token invalidation mitigates the chances of a data leak, which is a key NFR and a key risk we have taken into account

  1. SQL database vs NoSQL database

  1. Decision - SQL database (PostgreSQL)
  2. Rationale
  1. The inherent structure of Aunt Leah’s data (e.g clients in (1..many) programs, each client having (0..many) case notes) is relational.
  2. We can’t lose case notes or client data can’t be incorrect, therefore ACID compliance is a must
  3. Joins are required to get reports on KPIs, and to generate data visualisations as requested by David.
  1. Tradeoffs
  1. Lost
  1. Schema migrations are required for structural changes to the database, mitigated by prisma CLI
  1. Gained
  1. Data integrity guarantees and an ability to use powerful queries
  2. A SQL database plus a ORM like Prisma as outlined in our technical stack can save us time(estimated around 8-12 hours) by making the API layer less complex

  1. Auth: Self hosted vs SaaS (Clerk, Auth0)

  1. Decision - Self hosted BetterAuth
  2. Rationale
  1. Popular Auth SaaS solutions like Clerk do not have dedicated Canadian storage, which is a dealbreaker according to David
  2. We wanted to minimize how many enterprise solutions we used to reduce development costs, as well as simplify the future handover process as much as we can.
  1. Trade Offs
  1. Lost
  1. We would need to maintain the auth service ourselves, mitigated by BetterAuth’s excellent plugin support
  2. Increased application complexity and our workload (estimated 5-6 hours)
  1. Gained
  1. We ensure compliance to regulations like the eHealth Act in BC
  2. Zero ongoing costs

  1. Testing Strategy - Automated vs Manual Balance

  1. Decision - Automated Unit and Integration Tests but use User Acceptance Testing (UAT) for UI and E2E Testing
  2. Rationale
  1. Setting up automated UI tests using Storybook or Playwright is time consuming and the tests are tightly coupled to the UI
  2. Due to the iterative nature of this project, we expect our UI to change frequently as feedback from our clients and our peers come in
  3. However the backend and database requirements are not expected to change once development starts so we set a minimum 70% coverage on business logic, utilities, data transformations
  1. Tradeoffs
  1. Lost
  1. Not covering every single edge case, could lead errors in production that could have been caught
  1. Gained
  1. Increased velocity and less workload for our team.

  1. REST API vs gRPC

  1. Decision - REST API
  2. Rationale
  1. Most operations we need to support are CRUD operations
  2. REST JSON responses are sufficient for the scale of our application
  1. Tradeoff
  1. Lost
  1. Slower communication between microservices compared to gRPC's binary protocol
  2. Increased Payload size from JSON
  1. Gained
  1. Stateless API improves future scalability
  2. Broader ecosystem of tools, libraries, and integrations
  3. Our team is more familiar with REST APIs, allowing us to speed up development

  1. Reports Page

Higher Resolution: https://app.visily.ai/projects/49c24821-3755-4aac-a28f-2a026bf8660c/boards/2467296

Related User Story

  • Reporting (3.p): As a program director, I want to generate reports on program outcomes so that I can assess program effectiveness and report to funders.

Design Decisions

  • Pre-built reports: We wanted to support common metrics ready to go in one click. This will auto populate fields in the filter data section and if the user desires something different and unique, they can manually change the filter input fields.
  • Multi-criteria filtering: Although not all filters have been finalized, (we may add more in the future based on client feedback), our reports page delivers multi-criteria filtering so that users can get answers and insights into multiple questions in a faster time which improves workflow.
  • Unique individual count: Since our client mentioned the annoyance of having clients being double counted for certain workflows, we decided to have automatic deduplication across programs as a pre-built report.
  • Interactive visualizations: Switch between bar charts, pie charts, and line graphs to find best view which is individually exportable
  • Excel export: Export filtered data with selected fields for further analysis
  • Save Report Preset: If the user wants to save a report preset, they can click the “Save Preset” button which opens the above dialog box prompt for a name and description.

Key API Calls for Reports Page

  • On page load:
  • Call GET /api/saved-reports to load any saved report configurations (if user has saved reports)
  • When user selects a pre-built report card (e.g., "Client Breakdown"):
  • Call POST /api/reports with reportType: "client_breakdown" and filter values
  • Backend returns reportId and report data (chart + table)
  • When user changes filters and clicks "Apply Filter":
  • Call POST /api/reports with updated filters
  • When user switches chart type (Bar/Pie/Line):
  • No API call - frontend re-renders existing data with different visualization
  • When user clicks "Export Table As" or "Export Report As":
  • Call POST /api/reports/:reportId/export/excel ( or /pdf)
  • When user clicks "Save Preset":
  • Call POST /api/saved-reports with current report configuration
  • To load a previously saved report:
  • Call GET /api/saved-reports to get the configuration

Database Architecture

        Higher resolution: (https://dbdiagram.io/d/698a9646bd82f5fce23058ac)

Design Decisions

  • Since David has not given us the example data yet, we have attempted to create a bare bones database structure that we think we could build off easily once we receive the data we need
  • Use CUID (Collision-resistant Unique Identifiers) for all primary keys to prevent data leakage and to ensure url safety at the cost of using more space
  • Soft deletion used (is_active boolean flag) instead of hard deletion to undo easily and maintain a historical record of data at the cost of storing more data which might not be accessed a lot.
